# 変換正確性の優先度判断

## 適用条件

変換バグの優先順位を付けるとき、または TODO の項目を並び替えるとき。

## 制約

変換の問題を以下の 3 段階で分類し、上位の問題を常に先に解決する:

1. **サイレントに意味が変わる問題** — コンパイラが検出できず、実行時に誤った動作をする。最も危険
   - 例: switch の非リテラル case が変数バインディングになり、常にマッチする
   - 例: const が let mut にならず、TS では可変だった変数が Rust で不変になる
2. **コンパイルエラーになる問題** — 変換結果が使えないが、コンパイラが問題を検出する
   - 例: `.to_string()` が付かず型不一致エラー
   - 例: `NaN` が `f64::NAN` に変換されず未定義エラー
3. **未対応構文** — 変換自体が行われず、エラーメッセージまたはスキップで通知される
   - 例: `in` 演算子が未対応でエラー出力

理由: 意味の違う Rust コードを生成しても価値がない。コンパイルエラーは開発者が気づけるが、サイレントな意味相違は気づけない。

## 禁止事項

- 「小工数だから」という理由でサイレントな意味相違より先にコンパイルエラーの修正を優先すること
- 「影響範囲が広いから」という理由で未対応構文よりコンパイルエラーを後回しにすること
- 問題の深刻度（サイレント > コンパイルエラー > 未対応）を評価せずに、発見順や ID 順で優先度を決めること
