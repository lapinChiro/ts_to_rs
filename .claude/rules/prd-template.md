# PRD テンプレート

`backlog/` に配置する PRD は以下のフォーマットで記述すること。

```markdown
# <タイトル>

## 背景・動機

なぜこの機能が必要か。現状の課題や、この機能がないことで生じている問題。

## ゴール

この PRD が完了したとき、何ができるようになっているか。具体的・検証可能な形で書く。

## スコープ

### 対象

この PRD で実装する範囲を箇条書きで列挙する。

### 対象外

明示的に対象外とするものを列挙する。スコープクリープを防ぐ。

## 設計

### 技術的アプローチ

実装方針。既存のアーキテクチャとの関係、変更が必要なモジュール、新規追加するモジュール。

### 影響範囲

変更が及ぶファイル・モジュールの一覧。

## 作業ステップ

TDD で進める前提で、RED → GREEN → REFACTOR の順序を意識したステップに分解する。
各ステップは独立してテスト可能な単位にする。

- [ ] ステップ1: ...
- [ ] ステップ2: ...

## テスト計画

追加・変更するテストの概要。正常系・異常系・境界値を含む。

## 完了条件

この PRD の作業が「完了」と言える条件。品質チェック（clippy, fmt, test）を含む。
```

## 原則

- **着手可能であること**: PRD を読めば、追加の質問なしに実装を開始できる状態にする
- **スコープを絞ること**: 1つの PRD が大きくなりすぎたら分割する
- **YAGNI**: 将来の拡張を見越した設計を PRD に盛り込まない。今必要な範囲だけ設計する
