# TDD開発手順

新機能の追加・バグ修正は、以下の手順で実装すること:

1. **RED**: 失敗するテストを先に書く
2. **GREEN**: テストを通す最小限のコードを書く
3. **REFACTOR**: テストが通る状態を維持しながらリファクタする

テストを後から書くことは禁止。実装コードより先にテストを書け。

## テスト設計（REDの前に行うこと）

テストコードを書く前に、以下を明確にすること:

1. **検証項目の列挙**: 対象機能について何を検証すべきかをMECEに洗い出す
   - 正常系・異常系・境界値を漏れなく列挙する
2. **期待結果の定義**: 各項目について、入力に対してどのような出力・挙動が正しいかを決める
   - テスト名 `test_<対象>_<条件>_<期待結果>` がこれを表現する
3. テストコードは、この計画を機械的に記述したものであること

テストを「なんとなく書いて通ったから OK」にしない。
テストは仕様の表明であり、書く前に仕様が決まっていなければならない。

## Rustでの実践

- `#[cfg(test)] mod tests` にテストを追加してから実装を書く
- `cargo test -- <test_name>` で対象テストだけ実行して素早く RED → GREEN を回す
- REFACTOR フェーズでは `cargo clippy` も通すこと
