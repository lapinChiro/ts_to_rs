# バックログ管理

## 三層構造

| 場所 | 役割 |
|------|------|
| `plan.md` | ロードマップ。backlog/ と TODO の項目を**どの順序で・なぜその順で進めるか**を記載 |
| `backlog/` | 設計済み・着手可能な PRD。1機能 = 1ファイル |
| `TODO` | 未整理のアイデアの inbox。粒度不問 |

## フロー

1. 思いついたことは `TODO` に書く
2. 整理のタイミングで `TODO` の項目を精査する
3. 設計が十分にできる項目は PRD を書いて `backlog/<name>.md` に配置する
4. `backlog/` に移した項目は `TODO` から削除する
5. まだ曖昧・情報不足な項目は `TODO` に残す

## plan.md の更新タイミング

以下のイベントが発生したとき、`plan.md` の消化順序を見直すこと:

- `backlog/` にアイテムが**追加**されたとき — 新アイテムを順序リストの適切な位置に挿入
- `backlog/` のアイテムが**完了**したとき — 順序リストから削除し、完了を反映

## 完了したアイテムの扱い

- 作業が完了した `backlog/` の PRD ファイルは**削除**する
- 完了の記録は git history で追える。ファイルを残す必要はない
- `plan.md` の該当エントリも削除する

## backlog ファイルの命名

- ケバブケース: `import-export.md`, `directory-support.md`
- 機能の内容が名前から推測できること

## PRD の書き方

`backlog/` に配置する PRD は `.claude/rules/prd-template.md` のフォーマットに従うこと。
