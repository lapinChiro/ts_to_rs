# rustfmt 連携

## 背景・動機

現在、generator が出力する Rust コードは自前のインデント処理で整形しているが、生成コードが複雑になるにつれてフォーマットの一貫性を手動で維持するコストが増す。`rustfmt` を生成後に適用することで、常に標準的な Rust フォーマットの出力を保証できる。

また、generator 側のフォーマット処理を簡素化でき、新しい構文対応時の generator 実装が楽になる。

## ゴール

- CLI の出力する `.rs` ファイルが `rustfmt` でフォーマット済みの状態になる
- ライブラリの `transpile()` 関数の戻り値はフォーマット前のまま（副作用を持たせない）

## スコープ

### 対象

- CLI（`main.rs`）での単一ファイル出力後の `rustfmt` 適用
- CLI でのディレクトリ出力後の `rustfmt` 一括適用
- `rustfmt` が見つからない場合のフォールバック（警告を出してフォーマットなしで続行）

### 対象外

- ライブラリ API（`transpile()`）の変更
- `rustfmt` の設定ファイル（`rustfmt.toml`）の生成
- generator 内部のフォーマット処理の削除（段階的に進めるため今回はしない）

## 設計

### 技術的アプローチ

1. `main.rs` で `.rs` ファイル書き出し後に `rustfmt <path>` を `std::process::Command` で実行
2. `rustfmt` のインストール有無を `which rustfmt` または直接実行の結果で判定
3. `rustfmt` が見つからない / 失敗した場合は警告を `stderr` に出して続行（エラーにしない）

ディレクトリモードでは、全ファイル変換後にまとめて `rustfmt` を実行する（1ファイルずつではなく、対象ファイル一覧を引数で渡す）。

### 影響範囲

| ファイル | 変更内容 |
|----------|----------|
| `src/main.rs` | `rustfmt` 実行処理の追加 |
