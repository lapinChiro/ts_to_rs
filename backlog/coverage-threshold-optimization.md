# テストカバレッジ計測の最適化

## 背景・動機

CI のカバレッジ計測（`cargo llvm-cov --fail-under-lines 85`）に以下の問題がある:

1. **main.rs の汚染**: main.rs（CLI エントリポイント、行カバレッジ 29%）が全体を押し下げており、ライブラリコードの品質を正確に反映していない。全未カバー行 737 行のうち 117 行（15.9%）が main.rs
2. **閾値の乖離**: 実測 89.81% に対し閾値 85% では約 350 行分のカバレッジ喪失まで検知できず、「知らぬうちにカバレッジが下がる」ことを防げない

なお、本 PRD は `backlog/main-rs-refactoring.md`（main.rs リファクタリング）の完了後に実施する。リファクタリングによりビジネスロジックが lib.rs / directory.rs に移動するため、main.rs の性質が変わり、カバレッジの実測値も変動する。

## ゴール

- CI のカバレッジ閾値が実測値に対して適切なマージン（2pt 以内）に設定されている
- main.rs の低カバレッジがライブラリコードの品質指標を歪めていない
- テストを書かずにコードを追加した場合に CI が検知できる

## スコープ

### 対象

- `--ignore-filename-regex` による main.rs の計測除外
- リファクタリング後の実測値に基づく閾値の再設定
- `.github/workflows/ci.yml` の更新
- `CLAUDE.md` のカバレッジコマンド例の更新

### 対象外

- ファイル単位のカバレッジ閾値設定（`cargo llvm-cov` が未対応）
- Codecov 等の外部カバレッジサービスとの連携
- main.rs のテスト追加（リファクタリング後は I/O グルーのみとなるため不要）
- カバレッジメトリクスの変更（`--fail-under-lines` を継続使用。理由: 直感的・安定・ラチェット運用に適する）

## 設計

### 技術的アプローチ

#### 1. main.rs の除外

```bash
cargo llvm-cov --ignore-filename-regex 'main\.rs' --fail-under-lines <閾値>
```

#### 2. 閾値の決定手順

main.rs リファクタリング完了後に以下を実行:

1. `cargo llvm-cov --ignore-filename-regex 'main\.rs' --fail-under-lines 0` で実測値を取得
2. 実測値から 1〜2pt 引いた値を閾値に設定

リファクタリング前の参考値（main.rs 除外時）: 91.23%。リファクタリングにより lib.rs / directory.rs にテスト付きロジックが移動するため、同等以上になる見込み。

#### 3. 閾値の運用ルール

カバレッジが閾値を 2pt 以上上回ったら、閾値を 1pt 引き上げる（ラチェット）。この運用ルールは CLAUDE.md に記載する。

### 影響範囲

- `.github/workflows/ci.yml` — カバレッジ計測コマンドの変更
- `CLAUDE.md` — Key Commands セクションのカバレッジコマンド例の更新、運用ルールの追記
