# リファクタリング完了後の構造レビュー

## 背景・動機

3つの構造的リファクタリング PRD（モジュール再構成、IR改善+Registry 2パス化、型コンテキスト伝播）の完了後に、リファクタリングの品質を検証する体系的なレビューを実施する。

初回のアーキテクチャレビュー（`report/architecture-review.md`）で特定された問題が全て解消されていること、リファクタリングが新たな構造的問題を生んでいないことを保証する。

## ゴール

1. 初回レビューの全指摘事項が解消されていることを定量的に検証する
2. リファクタリングが新たに導入した問題がないことを検証する
3. レビュー結果を `report/post-refactoring-review.md` に文書化する

## スコープ

### 対象

- 全ソースファイル（src/、tests/）の構造レビュー
- 初回レビュー指摘事項の解消確認
- 新規の構造的問題の検出
- 変換ロジックの正確性検証（/correctness-audit の範囲）

### 対象外

- パフォーマンス最適化の実施

## 設計

### レビュー観点と定量的合格基準

#### ハイレイヤー: モジュール間の協調

| # | 観点 | 合格基準 | 検証コマンド |
|---|------|---------|------------|
| H1 | 依存方向の一貫性 | parser → registry/transformer → generator の方向に違反する `use` が 0 件 | `grep -rn "use crate::" src/` で依存グラフを構築し、逆方向依存を検出 |
| H2 | 循環依存の排除 | モジュールレベル（`src/transformer/` と `src/registry.rs` 間等）の循環 use が 0 件 | 依存グラフの閉路検出 |
| H3 | API 表面積の最小化 | `pub fn`（`pub(crate)` / `pub(super)` でない）が必要最小限 | `grep -c "^pub fn" src/**/*.rs` で計測し、外部から呼ばれない `pub fn` を検出 |

#### 中間レイヤー: モジュール内の構造

| # | 観点 | 合格基準 | 検証方法 |
|---|------|---------|---------|
| M1 | ファイルサイズ | 全ファイルが 600 行以下（テストファイルを除く） | `wc -l src/**/*.rs` |
| M2 | DRY | 15行以上のトークン同一コードブロックが 0 件 | 目視確認（主要な重複パターン: パラメータ変換、イテレータラッピング） |
| M3 | 疎結合 | サブモジュール間の `super::submodule::func` 直接呼び出しが各ファイルで 5 件以下 | `grep -c "super::" src/transformer/expressions/*.rs` |
| M4 | 直交性 | 以下の仮想変更要求が各1ファイルのみに影響: (a) 新しい文字列メソッド追加, (b) 新しい型変換追加 | 影響範囲分析 |
| M5 | 直交性（困難なケース） | 以下の仮想変更要求の影響ファイル数を計測し、現在の設計の限界を文書化: (a) ジェネリクス対応の追加, (b) ExprContext へのフィールド追加, (c) 新しい IR ノード追加 | 影響範囲分析 |

#### ローレイヤー: 実装の品質

| # | 観点 | 合格基準 | 検証コマンド |
|---|------|---------|------------|
| L1 | 関数サイズ | 100行超の関数が 0 件 | 全関数の行数計測 |
| L2 | コメント自己完結性 | `I-XX` 参照が src/ と tests/ に 0 件 | `grep -rn "I-[0-9]" src/ tests/` |
| L3 | TODO 妥当性 | 全 TODO コメントが TODO ファイルの項目と対応、または自己完結的 | `grep -rn "TODO" src/` で抽出し、TODO ファイルと照合 |
| L4 | エラーハンドリング | `unwrap()` / `expect()` がライブラリコード（tests 除外）に 0 件 | `grep -rn "\.unwrap()\|\.expect(" src/ --include="*.rs"` から test 除外 |
| L5 | 型安全性 | `RustType::Named` の name に `&` / `dyn ` / `Box<` が 0 件 | ソースコード内の `format!` パターン検索 |
| L6 | API 一貫性 | 同カテゴリの `convert_*` 関数が同一シグネチャパターンを持つ | 目視確認 |

### 初回レビュー指摘事項のチェックリスト

| # | 指摘事項 | 検証方法 |
|---|---------|---------|
| R1 | registry ↔ transformer の循環依存が解消 | `registry.rs` 内の `convert_ts_type` 呼び出しが空 registry でないことを確認 |
| R2 | expressions/mod.rs が分割されている | ファイル一覧とサイズで確認 |
| R3 | transformer/mod.rs の責務過多が解消 | クラス前処理・継承統合が classes.rs にあることを確認 |
| R4 | パラメータ変換ロジックの重複が排除 | `convert_function_param_pat` ヘルパーの存在を確認 |
| R5 | イテレータラッピングの重複が排除 | `build_iter_method_call` ヘルパーの存在を確認 |
| R6 | I-XX コメントが自己完結的説明に置換 | L2 で検証 |
| R7 | Named 型の文字列エンコードが構造的表現に置換 | `RustType::Ref` と `DynTrait` の存在を確認。L5 で検証 |
| R8 | register_interface の二重管理が解消 | `register_interface` メソッドが存在しないことを確認 |
| R9 | 型ヒント伝播の漏れが改善 | `ExprContext` の存在。カテゴリ B 改善の実装を確認 |

### レポートの構成

```markdown
# リファクタリング後の構造レビュー結果

**基準コミット**: `git rev-parse --short HEAD`
**レビュー日**: YYYY-MM-DD

## 1. 初回レビュー指摘事項の解消確認
R1〜R9 の各項目の合格/不合格と根拠

## 2. ハイレイヤー評価
H1〜H3 の各項目の計測結果と合格/不合格

## 3. 中間レイヤー評価
M1〜M5 の各項目の計測結果と合格/不合格
M5 は「不合格」ではなく「現在の設計の限界」として文書化

## 4. ローレイヤー評価
L1〜L6 の各項目の計測結果と合格/不合格

## 5. 新規発見事項
リファクタリングで新たに生じた問題（もしあれば）

## 6. 総合評価
全項目が合格: 構造的リファクタリング完了
不合格項目あり: TODO に記録し、対応方針を記載
```

### 影響範囲

| ファイル | 変更内容 |
|---------|---------|
| `report/post-refactoring-review.md` | 新規: レビュー結果レポート |
| `TODO` | 残存する問題の追記（もしあれば） |

## 作業ステップ

- [ ] 1: 全テスト PASS の前提確認
- [ ] 2: R1〜R9（初回レビュー指摘事項）の検証
- [ ] 3: H1〜H3（ハイレイヤー）の計測と検証
- [ ] 4: M1〜M5（中間レイヤー）の計測と検証
- [ ] 5: L1〜L6（ローレイヤー）の計測と検証
- [ ] 6: 新規発見事項の記録

- [ ] 8: 不合格項目があれば TODO に記録

## テスト計画

レビュータスクのため、テストの追加・変更は行わない。

## 完了条件

- [ ] 全レビュー観点（H1-3, M1-5, L1-6）の計測結果が文書化されている
- [ ] R1〜R9 の全項目が検証されている
- [ ] レビュー結果が `report/post-refactoring-review.md` に保存されている
- [ ] 「不合格」項目が 0 件、または対応する TODO が記録されている
