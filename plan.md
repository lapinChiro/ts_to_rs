# ts_to_rs 開発計画

## フェーズ1: 初版プロトタイプ — 完了

単一TSファイルの基本構造（interface, type, 関数, プリミティブ型, Optional, 配列, enum, テンプレートリテラル, export）をRustに変換するプロトタイプ。95テスト全パス。

## フェーズ2: 実用化 — 完了

#### ~~1. 生成コードのコンパイル検証~~ — 完了

全 fixture の生成コードが `rustc` でコンパイル可能であることを自動検証するテスト (`tests/compile_test.rs`) を追加。今後追加される fixture も自動的に検証対象になる。

#### ~~2. import/export 対応~~ — 完了

named import を `use crate::...` に変換、`export` の有無で `pub` / 非 `pub` を切り替え。外部パッケージ import はスキップ。

#### ~~3. ディレクトリ対応~~ — 完了

ディレクトリ入力時に再帰的に `.ts` ファイルを収集・変換し、`mod.rs` を自動生成。`node_modules/`・隠しディレクトリ・`.d.ts` は除外。

## フェーズ3: 機能拡充

### 消化順序

依存関係と価値の大きさに基づいて以下の順序で進める。

#### 1. CI — `backlog/ci.md`

**理由**: 全ての開発の品質ゲート。先にやることで以降の変更が安全になる。スコープが小さく即完了できる。

#### 2. rustfmt 連携 — `backlog/rustfmt-integration.md`

**理由**: CI と並行可能。生成コードの品質を底上げし、generator 拡張時のフォーマット負荷を減らす。スコープが小さい。

#### 3. クラス → struct + impl — `backlog/class-conversion.md`

**理由**: 実際の TS プロジェクトで最も頻出する未対応構文。対応可能な TS コードの範囲が大幅に広がる。

#### 4. ジェネリクス — `backlog/generics.md`

**理由**: クラスと独立して進められるが、クラスと組み合わせると価値が大きい。型システムの表現力を拡張する。

#### 5. クロージャ / 高階関数 — `backlog/closures.md`

**理由**: アロー関数は TS で多用される。クラス・ジェネリクスの後に対応することで、メソッド内のクロージャも含めた変換が可能になる。
