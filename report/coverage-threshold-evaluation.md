# テストカバレッジ閾値の妥当性調査

- **基準コミット**: `4a068cc`
- **未コミット変更**: なし（クリーン状態）

## 要約

現在の CI は `cargo llvm-cov --fail-under-lines 85` で全ファイル合算の行カバレッジを計測している。この方式には **2つの問題** がある:

1. **main.rs の汚染**: main.rs（CLI エントリポイント、29.09%）が全体を 1.4pt 押し下げており、ライブラリコードの品質を正確に反映していない
2. **閾値の乖離**: 実測 89.81% に対し閾値 85% では約 350 行分のカバレッジ喪失まで検知できない

**推奨**: main.rs を計測対象から除外し、閾値を **90%** に設定する。

## 分析 1: 計測対象 — main.rs を除外すべきか

### main.rs の性質

- 232 行（計測可能行 165 行）の CLI エントリポイント
- ファイル I/O、外部コマンド（`rustfmt`）呼び出し、ディレクトリ走査など副作用が中心
- ユニットテストでの網羅が本質的に困難（プロセス引数、ファイルシステム、外部コマンドに依存）
- ビジネスロジックはすべて `lib.rs` 以下に分離済みで、main.rs は薄いグルーコード

### main.rs が全体に与える影響

main.rs の 117 行の未カバーは全体の未カバー行 737 行の **15.9%** を占める。main.rs を含めると:

- ライブラリコードのカバレッジ改善努力が main.rs に希釈される
- main.rs に機能を追加するたびに全体カバレッジが下がり、閾値の意味が変わる
- 閾値を現在値に近づけられない（main.rs の変動を吸収するためにマージンが必要）

### 結論: main.rs を除外すべき

`--ignore-filename-regex main\.rs` を追加することで:

- ライブラリコードの品質を正確に計測できる
- 閾値を実測値に近づけられる（検出感度が上がる）
- main.rs への機能追加が閾値を不安定にすることがなくなる

main.rs のテストが完全に不要というわけではないが、カバレッジ閾値による品質ゲートの対象としては不適切。main.rs のロジックが複雑化した場合は、ロジックを lib.rs 側に移動して間接的にテストするのが正しいアプローチ。

## 分析 2: メトリクス — 何を閾値にするか

`cargo llvm-cov` は3種類のメトリクスを提供する:

| メトリクス | 意味 | 現在値（main.rs 除外） |
|---|---|---|
| `--fail-under-lines` | 実行された行の割合 | 91.23% |
| `--fail-under-functions` | 1回以上実行された関数の割合 | 90.64% |
| `--fail-under-regions` | 実行されたコードリージョン（分岐等）の割合 | 89.73% |

さらに、割合ではなく絶対数で指定するオプションもある:

| オプション | 意味 | 現在値（main.rs 除外） |
|---|---|---|
| `--fail-uncovered-lines` | 未カバー行の上限 | 620 |
| `--fail-uncovered-functions` | 未カバー関数の上限 | 48 |

### 各メトリクスの比較

| 観点 | `--fail-under-lines` | `--fail-under-regions` | `--fail-uncovered-lines` |
|---|---|---|---|
| 直感性 | 高い。「全体の何%をテストしているか」が分かりやすい | 低い。リージョンの定義がコンパイラ依存 | 中程度。「未テスト行が何行あるか」 |
| 検出精度 | 中。コード追加時に分母が増えるため、少量の未テストコードが希釈される | 高。分岐レベルで計測するため、分岐網羅の低下を検出しやすい | 高。コード追加時にテストなしなら数値が直接増える |
| 安定性 | 高。コードベース全体に対する割合なので変動が小さい | 中。リファクタリングでリージョン数が変わると変動する | 低。コード追加のたびに調整が必要 |
| ラチェット適性 | 高。実測値に基づいて閾値を引き上げやすい | 中 | 低。コードベースの成長に合わせて定期的に再設定が必要 |

### 結論: `--fail-under-lines` が適切

- 最も直感的で、チームメンバー（将来含む）が理解しやすい
- 安定性が高く、通常の機能追加でフラップしにくい
- ラチェット運用に適している
- 現在の方式からの変更が最小限

`--fail-uncovered-lines` はコード追加時のテスト漏れ検出に優れるが、コードベースの成長に合わせて閾値を頻繁に再設定する必要があり、運用コストが高い。

`--fail-under-regions` は分岐網羅の精度が高いが、リージョンの定義がコンパイラ依存で直感的でなく、リファクタリングで数値が変動しやすい。

## 分析 3: 閾値 — いくつに設定するか

main.rs 除外後の実測値 **91.23%** を基準に検討する。

### 閾値の選択肢

| 選択肢 | 閾値 | マージン | 行数換算（未カバー許容増分） | 特徴 |
|---|---|---|---|---|
| A: タイト | **91%** | 0.23pt | 約 16 行 | ほぼ全てのテスト漏れを即座に検知。ただし小さな機能追加でも閾値を割る可能性がある |
| B: 推奨 | **90%** | 1.23pt | 約 87 行 | テスト漏れを早期に検知しつつ、通常の開発で閾値を割りにくい |
| C: やや緩め | **89%** | 2.23pt | 約 158 行 | 安定運用しやすいが、検出が遅れる |

### マージンの妥当性分析

このプロジェクトの典型的な機能追加（TODO の Tier 1 項目）は 50〜150 行程度の追加になることが多い。テスト付きで追加する場合、テスト対象コードのカバレッジは通常 90% 以上になるため、全体のカバレッジはほぼ維持される。

カバレッジが下がるケース:
- テストなしでコードを追加した場合（検知したいケース）
- エラーハンドリングの分岐が増えた場合（軽微な低下、許容すべき）

**90%（マージン 1.23pt）** であれば:
- テストなしで 87 行以上のコードを追加すると検知される → 十分な感度
- エラーハンドリング追加程度の軽微な低下は許容される → 偽陽性が少ない
- 今後カバレッジが上がれば 91% に引き上げてラチェットできる

## 推奨案

### CI コマンドの変更

```bash
# Before
cargo llvm-cov --fail-under-lines 85

# After
cargo llvm-cov --ignore-filename-regex 'main\.rs' --fail-under-lines 90
```

### 変更箇所

1. `.github/workflows/ci.yml` — カバレッジ計測コマンドの変更
2. `CLAUDE.md` — Key Commands セクションのカバレッジコマンド例の更新

### 運用ルール

- カバレッジが実測値より 2pt 以上上がったら、閾値を 1pt 引き上げる（ラチェット）
- main.rs のロジックが複雑化した場合は、ロジックを lib.rs 側に移動してテスト対象にする
