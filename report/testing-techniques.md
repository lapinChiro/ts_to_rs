## 1. ブラックボックステスト技法

ブラックボックステスト（仕様ベーステスト / specification-based testing）は、内部実装を参照せず、仕様・インターフェースのみからテストケースを導出する。ISTQB CTFL v4.0 では "specification-based techniques" として定義されている。

---

### 1.1 同値分割（Equivalence Partitioning）

**定義と目的**

入力値（または出力値）を「同じ処理が適用される」グループ（等価クラス）に分割し、各クラスから代表値を1つ選んでテストする技法。同じクラス内の値は同じ種類の欠陥を検出すると仮定する。

- **有効パーティション**: 仕様上受理される入力のグループ
- **無効パーティション**: 仕様上拒否される入力のグループ
- 各値は必ず1つのパーティションにのみ属する

**適用場面**

入力の種類が多く全数テストが非現実的な場合。数値範囲、文字列種別、型バリアント等。

**コンパイラ/トランスパイラへの適用例**

```
TypeScript の数値型変換:
  - 整数リテラル (0, 1, -1, 100) → i32/i64 パーティション
  - 浮動小数点リテラル (1.5, -0.1) → f64 パーティション
  - NaN, Infinity → f64::NAN, f64::INFINITY パーティション
  - 大きすぎる整数 → i128 or 変換エラーパーティション

TypeScript の文字列型:
  - リテラル文字列 "foo" → &str / String パーティション
  - テンプレートリテラル `${x}` → format!() パーティション
  - 文字列メソッド呼び出し → .to_string() / .as_str() パーティション
```

**フィクスチャ適用例**: `basic-types.input.ts`, `string-methods.input.ts`

**限界**

- パーティション内部の差異を見落とす
- 複数パーティションの組み合わせ（デシジョンテーブル）が必要な場合に対応しない
- **補完**: 境界値分析（境界での挙動を追加）、ペアワイズ（組み合わせ）

---

### 1.2 境界値分析（Boundary Value Analysis）

**定義と目的**

同値分割の拡張。順序付けされたパーティションの境界付近でバグが最も多く発生するという経験則に基づき、境界値とその近傍を重点テストする。

ISTQB CTFL v4.0 の2種類:
- **2値BVA**: 境界値そのものと、隣接パーティションの最近傍値（各境界に2テスト）
- **3値BVA**: 境界前・境界・境界後（各境界に3テスト）

**適用場面**

数値範囲、文字列長、配列サイズ等、順序付けが意味を持つ入力。

**コンパイラ/トランスパイラへの適用例**

```
整数型の境界:
  i32::MAX (2147483647) の前後: 2147483646 / 2147483647 / 2147483648
  i32::MIN (-2147483648) の前後: -2147483649 / -2147483648 / -2147483647
  → オーバーフロー時の変換挙動（i64へ昇格するか、エラーか）

配列/タプルの境界:
  空配列 [] → Vec::new()
  1要素 [x] → vec![x]
  大量要素 → パフォーマンス上の境界

ネスト深さの境界:
  関数ネスト深さ: 再帰上限への配慮
  型のネスト深さ: Option<Option<Vec<T>>> 等
```

**フィクスチャ適用例**: `array-literal.input.ts`, `update-expr.input.ts`

**限界**

- 順序付けのない入力（ブール、列挙型）には適用不可
- 複数の境界が相互作用する場合を網羅しない
- **補完**: 同値分割（パーティション内代表値）、デシジョンテーブル（組み合わせ境界）

---

### 1.3 デシジョンテーブル（Decision Table）

**定義と目的**

複数の条件とその組み合わせに対するアクション（出力）を表形式で網羅的に表現する技法。N条件の場合、最大 2^N 列の組み合わせを生成するが、実際には到達不可能な組み合わせを除去して簡約化する。

**構成要素**
- **条件行**: テスト対象となる論理条件
- **アクション行**: 各条件組み合わせに対するシステムの動作
- **ルール列**: 条件の各組み合わせ（1列 = 1テストケース）

**適用場面**

複数の独立した条件が絡み合うビジネスルール。型変換ルールの優先度決定、型チェックの複合条件。

**コンパイラ/トランスパイラへの適用例**

```
関数パラメータの変換ルール:
  条件1: デフォルト値あり → Option<T> + .unwrap_or()
  条件2: オプショナル(?) → Option<T>
  条件3: レスト引数(...) → Vec<T>

  | デフォルト値 | オプショナル | レスト | 変換先              |
  |------------|------------|------|-------------------|
  | Y          | Y          | N    | Option<T> w/ default|
  | Y          | N          | N    | T w/ default (or Option)|
  | N          | Y          | N    | Option<T>         |
  | N          | N          | Y    | Vec<T>            |
  | N          | N          | N    | T                 |

型narrowingの条件組み合わせ:
  条件1: null check (x !== null)
  条件2: typeof check (typeof x === "string")
  条件3: instanceof check (x instanceof Foo)
  → 各組み合わせの生成コード確認
```

**フィクスチャ適用例**: `default-params.input.ts`, `type-narrowing.input.ts`, `optional-fields.input.ts`

**限界**

- 条件数が多いと組み合わせが爆発的に増加（2^N問題）
- 条件間の依存関係表現が難しい
- **補完**: ペアワイズテスト（組み合わせ爆発の緩和）

---

### 1.4 状態遷移テスト（State Transition Testing）

**定義と目的**

有限状態機械（FSM: Finite State Machine）のモデルに基づき、状態・イベント・遷移・アクションを網羅するテストケースを設計する技法。

**表現形式**
- **状態遷移図**: 状態をノード、遷移をエッジとするグラフ
- **状態遷移表**: 状態×イベントのマトリクス（無効遷移も明示）

**カバレッジ基準**
- **全状態カバレッジ**: 全状態を最低1回通過
- **全遷移カバレッジ**: 全有効遷移を最低1回通過
- **全遷移ペアカバレッジ**: 連続する全遷移ペアをカバー

**適用場面**

フェーズを持つ処理系、状態を追跡するオブジェクト。

**コンパイラ/トランスパイラへの適用例**

```
TypeCollector の状態遷移:
  状態: 初期 → 収集中 → 解決待ち → 完了
  イベント: 型宣言発見 / 参照解決 / 全ファイル処理完了

  遷移テスト:
  - 前方参照（宣言前使用）→ 解決待ち状態が正しく機能するか
  - 循環参照 → 無限ループしないか
  - 空ファイル → 初期→完了への直接遷移

Scopeスタックの状態:
  状態: グローバルスコープ / 関数スコープ / ブロックスコープ
  イベント: 関数入場・退場 / ブロック入場・退場

  テスト:
  - ネストしたスコープからの変数参照
  - クロージャによるスコープキャプチャ
  - 早期return後のスコープ解放
```

**フィクスチャ適用例**: `closures.input.ts`, `type-registry.input.ts`, `class-inheritance.input.ts`

**限界**

- 状態空間が大きい場合、全遷移カバレッジが非現実的
- 並行状態・ヒストリー状態等の拡張FSMには不十分
- **補完**: 同値分割（状態ごとの入力クラス）

---

### 1.5 ユースケーステスト（Use Case Testing）

**定義と目的**

システムがどのように使われるかという「ユースケース」に基づき、エンドツーエンドのシナリオからテストケースを導出する。ユーザーの実際の利用フローを忠実にトレースする。

**構成要素**
- **基本フロー（Main/Happy Path）**: 最も典型的な成功シナリオ
- **代替フロー（Alternative Flows）**: 正常だが非典型的なシナリオ
- **例外フロー（Exception Flows）**: エラー・失敗シナリオ

**適用場面**

CLIツール、API、エンドユーザー向け機能のテスト。

**コンパイラ/トランスパイラへの適用例**

```
ユースケース: TypeScriptのHonoプロジェクトをRustに変換する

基本フロー（Happy Path）:
  1. hono/src/hono.ts を入力
  2. 全インポートが解決される
  3. クラス・関数・型が変換される
  4. 出力Rustがcargo checkを通過
  5. E2Eで実行等価性を確認

代替フロー:
  - 一部ファイルのみ変換（--include フィルタ）
  - エラーレポートモード（--report-unsupported）
  - ディレクトリ変換 vs 単一ファイル変換

例外フロー:
  - 構文エラーを含むTSファイル
  - 未対応構文（変換不可能なTS固有機能）
  - 循環import
  - 存在しないファイルパス
```

**フィクスチャ適用例**: `tests/e2e_test.rs`, `tests/cli_test.rs`, Honoベンチマーク

**限界**

- ユースケースは高レベルすぎて詳細な欠陥を見逃しやすい
- 内部ロジックのカバレッジ保証なし
- **補完**: 同値分割・境界値分析（ユースケースの入力部分を精緻化）

---

### 1.6 ペアワイズテスト（All-Pairs Testing）

**定義と目的**

複数のパラメータが存在するとき、すべてのパラメータの**ペア（2値の組み合わせ）**を少なくとも1回カバーするテストケースセットを生成する。研究によれば、大部分の欠陥は1〜2個のパラメータの相互作用に起因するため、全組み合わせの代わりにペアカバレッジで十分な欠陥検出率を得られる。

**効果**
- 全組み合わせ: 10パラメータ × 各3値 = 59,049テスト
- ペアワイズ: 同条件で約 15〜25 テストに削減（95%以上削減）
- 欠陥検出率: 全組み合わせと比較して高い実績

**ツール**
- Microsoft PICT（Pairwise Independent Combinatorial Tool）: OSS、高速
- 50パラメータ × 20値でも20秒未満で生成

**適用場面**

多数の設定パラメータ、フラグの組み合わせ、複数の型特性の交差。

**コンパイラ/トランスパイラへの適用例**

```
型変換のパラメータ:
  P1: オプショナリティ {必須, オプショナル(?), デフォルト値あり}
  P2: 型種別 {プリミティブ, ユニオン, ジェネリック, 配列}
  P3: スコープ {グローバル, ローカル, クロージャキャプチャ}
  P4: 可変性 {const, let}

  全組み合わせ: 3×4×3×2 = 72
  ペアワイズ: 約12〜15テストで全ペアを網羅

関数シグネチャの組み合わせ:
  P1: 引数の型 {string, number, boolean, any, T}
  P2: 戻り値型 {void, T, Promise<T>, T|null}
  P3: async/sync {同期, 非同期}
  P4: アクセス修飾子 {public, private, protected}（クラスメソッド）
```

**限界**

- 3パラメータ以上の複合相互作用（3-way interaction）による欠陥を見逃す可能性
- テストケース生成ツールへの依存
- **補完**: 専用の3-way以上の組み合わせテスト（リスクの高い部分のみ）

### 1.7 エラー推測（Error Guessing）

**定義と目的**

テスターの経験・直感・ドメイン知識に基づき、エラーが起きやすい箇所を予測してテストケースを設計する。正式な技法ではなく「欠陥催促（fault attack）」とも呼ばれる。

**典型的なエラー推測リスト**
- 空値（null, undefined, "", [], {}）
- 境界値（0, -1, MAX, MIN）
- 特殊文字（\n, \t, null-byte, Unicode surrogate pair）
- 重複・循環（循環参照、重複宣言）
- 競合・優先順位の衝突

**コンパイラ/トランスパイラへの適用例**

```
TypeScript固有の罠（Error Guessing対象）:
  - undefined vs null の混在
    input: x === undefined  →  期待: Option比較 or != null判定
  - NaN の伝播
    input: NaN + 1  →  生成: f64::NAN + 1.0（f64で維持）
  - 型強制（Type Coercion）
    input: "5" + 3  →  意味的変換 or 変換エラー?
  - typeof null === "object" のTS固有挙動
  - 0 == false / "" == false の緩やかな等価性
  - Number("") === 0 の暗黙変換
  - プロトタイプチェーン上のメソッド継承
  - TS enum の数値/文字列の二面性
  - declare const の環境変数注入パターン
  - namespace / module パターン
  - オーバーロードシグネチャ
```

**フィクスチャ適用例**: `unsupported-syntax.input.ts`, `union-type.input.ts`, `conditional-type.input.ts`

**限界**

- テスターのスキルに大きく依存
- 体系的な網羅性の保証なし
- **補完**: チェックリストベーステスト（エラー推測を体系化）

## 2. ホワイトボックステスト技法

ホワイトボックステスト（構造ベーステスト / structure-based testing）は、コードの内部構造（制御フロー、データフロー）を分析してテストケースを設計する。カバレッジ基準が客観的な指標となる。

---

### 2.1 命令網羅（Statement Coverage / C0）

**定義と目的**

プログラム内のすべての実行可能な命令（ステートメント）を少なくとも1回実行するテストケースを設計する。最も基本的かつ最弱のカバレッジ基準。

**計算式**
```
Statement Coverage = (実行された命令数 / 全命令数) × 100%
```

**適用場面**

カバレッジの最低限ベースライン。高コスト・高リスク領域の特定に使う。

**本プロジェクトでの位置付け**

`cargo llvm-cov --fail-under-lines 89` で89%の行カバレッジ閾値を設定。llvm-cov の "lines" は実質 C0 に近い。

**限界**

- 分岐が未テストでも100%達成できる（if 文の else 側を踏まなくても良い）
- 真の欠陥検出率は低い
- **補完**: 分岐網羅（C1）が必須

---

### 2.2 分岐網羅（Branch Coverage / C1）

**定義と目的**

制御フローグラフの全エッジ（分岐）を少なくとも1回通過するテストケースを設計する。if/else 両方、switch の全 case、ループの入場・スキップ等をカバーする。

**C0との関係**

C1 ⊇ C0（分岐網羅を達成すれば命令網羅も達成）

**適用場面**

条件分岐を持つあらゆるロジック。実用的な最低カバレッジ基準として広く採用。

**コンパイラ/トランスパイラへの適用例**

```
transformer/expressions.rs のパターンマッチ:
  match expr {
    Expr::Lit(lit) => ...    // ← 必ずテストが必要
    Expr::Ident(ident) => ... // ← 必ずテストが必要
    Expr::Call(call) => ...   // ← 必ずテストが必要
    Expr::Member(member) => ...
    _ => unreachable!()       // ← unreachable! に到達しないことのテスト
  }

TypeResolver の型解決分岐:
  if resolved_type.is_some() { ... }  // Some/None 両方
  else { ... }
```

**限界**

- 複合条件（`a && b`）の個々の条件の真偽組み合わせを網羅しない
- **補完**: 条件網羅（C2）またはMC/DC

---

### 2.3 条件網羅（Condition Coverage / C2）

**定義と目的**

複合条件式内の各原子条件（atomic condition）の真偽を独立してカバーする。ただし、分岐の結果（全体として true/false）のカバレッジは保証しない。

**C1との関係**

C2 と C1 はいずれか一方が他方を包含しない（互いに独立）。C2 単独では分岐の真側・偽側が両方カバーされるとは限らない。

**適用場面**

複合条件ロジックの詳細検証。

**コンパイラ/トランスパイラへの適用例**

```
型チェックの複合条件:
  if is_optional && has_default_value {
    // 条件A(is_optional)の真偽 × 条件B(has_default_value)の真偽
    // C2: (T,T), (T,F), (F,T), (F,F) をカバー
  }
```

**限界**

- 分岐カバレッジを保証しない（これが主要な批判点）
- **補完**: MC/DC（条件独立性と分岐カバレッジを同時に保証）

---

### 2.4 MC/DC（Modified Condition/Decision Coverage）

**定義と目的**

1. 各決定（decision）のすべての結果（true/false）をテスト（= C1の要件）
2. 各原子条件のすべての値をテスト（= C2の要件）
3. **各原子条件が決定結果に独立して影響することを示す**（MC/DC固有の要件）

条件3が最重要: ある条件Xについて「他の条件を固定したまま X のみ変えたときに決定結果が変わる」テストペアが存在することを証明する。

**テスト数の上限**

N 個の原子条件を持つ決定について、MC/DC達成に必要なテスト数は **N+1** 以下（最悪 2N）。2^N の全組み合わせと比べ指数的に少ない。

**適用基準**

- DO-178B/DO-178C（航空ソフトウェア）: Level A（最高安全性レベル）で必須
- ISO 26262（自動車機能安全）: ASIL D で高推奨
- NASA NPR 7150.2D: 安全重要コンポーネントで必須

**コンパイラ/トランスパイラへの適用例**

```
TypeResolver における型解決条件:
  if (is_union_type && has_type_guard) || is_narrowed {
    apply_narrowing()
  }

  原子条件: A=is_union_type, B=has_type_guard, C=is_narrowed
  決定式: (A && B) || C

  MC/DC テストペア（Cが独立影響を示す）:
    A=T, B=T, C=F → 決定=T
    A=T, B=T, C=T → 決定=T  （Cを変えても結果変わらない → Aを変える等の工夫が必要）

  独立対の構成例（3条件 → 4テスト）:
    (T,T,F) → T
    (T,T,T) → T  (Cの独立性: (T,T,F)→T vs (T,T,T)→T ... 変わらない = Cの寄与なし)
    (F,F,T) → T  vs  (F,F,F) → F  → Cが独立影響を持つことを示す
    (T,F,F) → F  vs  (T,T,F) → T  → Bが独立影響を持つことを示す
    (F,T,F) → F  vs  (T,T,F) → T  → Aが独立影響を持つことを示す
```

**限界**

- テストケース設計が複雑（ツール支援が有効）
- トランスパイラの論理ロジックには有効だが、AST構造のテストには直接適用しにくい
- **補完**: スナップショットテスト（AST構造）+ MC/DC（変換判定ロジック）

---

### 2.5 パス網羅（Path Coverage）

**定義と目的**

制御フローグラフにおける全実行パスをカバーするテストケースを設計する。最も強力なホワイトボックス基準だが、ループの存在により実用的には無限パスが生じる。

**McCabe のサイクロマティック複雑度**

```
CC = E - N + 2P
  E: エッジ数
  N: ノード数
  P: 接続成分数（通常1）
```

- CC = 線形独立パス数の上界 = ベーシスパス数
- CC ≤ 10 が推奨（それ以上は関数分割を検討）
- **ベーシスパステスト**: CC 個のパスのみをカバー（パス網羅の実用的近似）

**適用場面**

高リスク・高複雑度の関数。型変換の中核ロジック。

**コンパイラ/トランスパイラへの適用例**

```
transformer/expressions.rs の transform_call_expr():
  CC が高い場合 → ベーシスパス抽出 → 各パスをカバーするフィクスチャを設計

TypeResolver::resolve_type():
  パス1: キャッシュヒット → 即座に返す
  パス2: ローカルスコープで解決
  パス3: 型レジストリで解決
  パス4: 未解決 → unknown フォールバック
```

**限界**

- ループがある場合は完全なパス網羅が不可能
- CC が高いコードでは非現実的なテスト数になる
- データの組み合わせによる意味的差異を捉えられない
- **補完**: データフローテスト（データの流れを追加考慮）

---

### 2.6 データフローテスト（Data Flow Testing）

**定義と目的**

変数の「定義（definition）」と「使用（use）」のペア（du-pair）に着目し、変数ライフサイクルの問題（未定義使用、定義済み未使用、多重定義等）を検出する技法。

**du-pair の種類**
- **all-defs**: 各定義から少なくとも1つの使用に到達するパスをカバー
- **all-uses**: 各定義から全ての使用に到達するパスをカバー（より強い）
- **all-du-paths**: 全定義から全使用への全パスをカバー（最強）

**検出対象の問題**
- 定義済み未使用変数（dead code）
- 未定義変数の使用
- 変数の多重定義（前の定義が上書き）
- 変数の解放後使用

**適用場面**

型収集・型解決フェーズの変数・型変数の正確な伝播確認。

**コンパイラ/トランスパイラへの適用例**

```
TypeCollector の型変数ライフサイクル:
  定義: collect_type_decl() で TypeRegistry に登録
  使用: resolve_type_ref() での参照

  du-pair テスト:
  - 前方宣言（use before def）: resolve_type_ref が先に呼ばれ、後で collect_type_decl
    → 2パスアルゴリズムで解決されているか
  - 型の上書き（重複定義）: 同名型の再定義時の挙動
  - 未使用型定義: TypeRegistry に登録されたが一度も参照されない型

expected_type（期待型）の伝播:
  定義: sink_expected_type() で親ノードから伝播
  使用: resolve_expr_type() での型強制

  du-pair テスト:
  - 期待型が正しく子ノードへ伝播するか
  - 期待型が存在しない場合のフォールバック
```

**フィクスチャ適用例**: `type-registry.input.ts`, `type-infer-unannotated.input.ts`, `assignment-expected-type.input.ts`

**限界**

- 解析ツールなしで手動適用するには複雑
- 動的型付けやポリモーフィズムに弱い
- **補完**: 静的解析ツール（clippy unused warnings 等）との組み合わせ

## 3. 経験ベーステスト技法

---

### 3.1 エラー推測（Experience-based）

1.7 節で詳述。ここでは経験ベース技法としての位置付けを整理する。

**分類**:「欠陥催促（fault attack）」とも呼ばれ、ISTQB では経験ベーステスト技法の一つ。

---

### 3.2 探索的テスト（Exploratory Testing）

**定義と目的**

テスト設計・実行・学習を並行して行い、リアルタイムに戦略を調整する。「スクリプテッドテスト（事前設計）の反対」ではなく、「スクリプトに縛られず思考を使うテスト」として定義される（James Bach）。

**セッションベーステスト管理（SBTM）**

Jonathan & James Bach（2000年）が提唱。構成要素:

- **チャーター（Charter）**: セッションのゴール・議題（例: "型narrowingの状態遷移を調査する"）
- **セッション**: 中断なしの1〜2時間の集中テスト期間
- **デブリーフィング**: セッション後の報告（発見バグ、カバレッジ、問題）

**ヒューリスティクス（探索テストの指針）**

- **SFDIPOT**: Structure, Function, Data, Interface, Platform, Operations, Time
- **HICCUPPS**: History, Image, Comparable products, Claims, Users, Product, Purpose, Statutes
- **リスクベース**: 過去のバグが多かった領域を重点的に

**コンパイラ/トランスパイラへの適用例**

```
チャーター例:
  1. "Honoプロジェクトのクラス継承パターンを変換し、
     Rustで意味が変わるケースを発見する"
     → 時間: 90分
     → 目標: 実行等価性が崩れるケースを3件以上特定

  2. "TypeScriptのunion型に関連する全変換コードパスを探索し、
     型情報が欠落するケースを特定する"
     → 時間: 60分
     → ツール: scripts/inspect-errors.py + hono-bench

  3. "async/await の変換シナリオを体系的に試み、
     panic または誤変換を発見する"
```

**Honoベンチマークとの関係**

`./scripts/hono-bench.sh` は一種の大規模探索的テスト。実際の大規模OSS（Hono）を使い、変換失敗パターンを発見する。`scripts/inspect-errors.py` でカテゴリ分析することが、「デブリーフィング」に相当する。

**限界**

- 体系的なカバレッジ保証なし
- スキルに強く依存
- 結果の再現が難しい（セッションの記録が重要）
- **補完**: スクリプテッドテスト（発見された欠陥の回帰テスト化）

---

### 3.3 チェックリストベーステスト（Checklist-based Testing）

**定義と目的**

過去の経験・業界標準・エラー推測から体系化した観点リストに基づいてテストを設計する。探索的テストとエラー推測の中間的な技法。

**特徴**
- 再利用可能な観点集
- ドメイン知識の形式化
- 個人スキルへの依存を軽減

**コンパイラ/トランスパイラ向けチェックリスト**

```
TypeScript AST ノード種別チェックリスト:
  □ Stmt::Decl::Var （let/const/var）
  □ Stmt::Decl::Fn （function宣言）
  □ Stmt::Decl::Class
  □ Stmt::Decl::TsInterface
  □ Stmt::Decl::TsTypeAlias
  □ Stmt::Decl::TsEnum
  □ Stmt::Decl::TsModule
  □ Expr::Lit（文字列/数値/bool/null）
  □ Expr::Ident
  □ Expr::Call
  □ Expr::Member
  □ Expr::Arrow
  □ Expr::Bin（演算子全種）
  □ Expr::Unary（!, -, typeof, void）
  □ Expr::Assign
  □ Expr::Cond（三項演算子）
  □ Expr::New
  □ Expr::Array
  □ Expr::Object

型システムチェックリスト:
  □ プリミティブ型（string, number, boolean, null, undefined, any, unknown, never）
  □ 配列型（T[], Array<T>）
  □ タプル型（[T, U]）
  □ オブジェクト型（{key: T}）
  □ 関数型（(a: T) => U）
  □ ユニオン型（T | U）
  □ インターセクション型（T & U）
  □ ジェネリック型（T, T extends U）
  □ 条件型（T extends U ? V : W）
  □ マップ型（{[K in keyof T]: V}）
  □ テンプレートリテラル型
  □ typeof 演算子（型レベル）
  □ keyof 演算子
  □ インデックスアクセス型（T[K]）

変換品質チェックリスト:
  □ 型安全性（生成コードがcargo checkを通過）
  □ 意味的等価性（E2Eで実行結果が一致）
  □ 未使用import/変数の警告なし
  □ clippy警告なし
  □ パニックしないこと（unwrap on None 等）
  □ 循環参照の安全な処理
```

**限界**

- リストが陳腐化しやすい（定期的な更新が必要）
- 網羅的でないことがある（リストにない問題は見逃す）
- **補完**: 探索的テスト（新しい観点の発見）、エラー推測（深掘り）

## 4. 構造的テスト技法 — コンパイラ/トランスパイラ特有

---

### 4.1 スナップショットテスト（Golden Master Testing）

**定義と目的**

プログラムの出力（テキスト、JSON、AST表現等）を初回実行時に「スナップショット」として保存し、以降の実行でスナップショットとの差分を検出する回帰テスト技法。テキスト変換プログラム（コンパイラ、トランスパイラ、シリアライザ等）に特に適している。

**採用ツール: insta（Rust）**

```rust
// 基本的な使い方
#[test]
fn test_basic_conversion() {
    let input = r#"const x: number = 42;"#;
    let output = transpile(input).unwrap();
    insta::assert_snapshot!(output);
}

// YAML形式で構造化データも扱える
insta::assert_yaml_snapshot!(ast_node);
```

**ワークフロー**
1. 初回: `cargo test` → "snapshot not found" で失敗
2. `cargo insta review` → diff を確認して承認/却下
3. 承認後: `tests/snapshots/test_name.snap` に保存
4. 以降: スナップショットと差分があればテスト失敗

**本プロジェクトでの活用**（`tests/integration_test.rs`）

```rust
snapshot_test!(test_basic_types);          // transpile()
snapshot_test!(test_type_narrowing, builtins); // transpile_with_builtins()
snapshot_test!(test_async_await, collecting);  // transpile_collecting()
```

**適用戦略**

- **ASTノード種別ごとのスナップショット**: 1つのフィクスチャが1〜3種類のASTパターンをカバー
- **リグレッション検出**: 既存の変換ロジック変更時に意図しない出力変化を即座に検出
- **仕様としての機能**: スナップショットが変換仕様のドキュメントになる

**限界**

- スナップショットが「正しい」という保証がない（初回の承認が重要）
- 大きなスナップショット差分は読みづらい
- **補完**: コンパイルテスト（生成コードが型安全か）、E2Eテスト（実行等価性）

---

### 4.2 コンパイル確認テスト（Compile-Check Testing）

**定義と目的**

生成されたRustコードを実際にコンパイラ（`cargo check`）にかけ、型エラー・構文エラーがないことを確認する技法。スナップショットテストが「生成物の形」を確認するのに対し、コンパイルテストは「生成物の型安全性」を確認する。

**本プロジェクトでの実装**（`tests/compile_test.rs`）

```
TS input → transpile() → Rust source → cargo check → pass/fail
```

- シングルファイルテスト: `tests/compile-check/src/lib.rs` に書き込んでcheck
- マルチファイルテスト: `tests/fixtures/multi/` の複数ファイルを同時check
- `COMPILE_LOCK` による直列化（共有プロジェクトへの同時書き込み防止）

**適用戦略**

- 新しい変換パターン追加時には対応するコンパイルテストを必ず追加
- コンパイルエラーの種類別カテゴリ化（型不一致、未解決パス、生存期間エラー）

---

### 4.3 実行等価性テスト（Differential / E2E Testing）

**定義と目的**

TypeScript実行（tsx）とRust実行の**出力が一致すること**を確認する差分テスト。変換が型安全であることだけでなく、**意味的等価性**を検証する最重要テスト層。

**本プロジェクトでの実装**（`tests/e2e_test.rs`）

```
TS input  → tsx 実行 → stdout A
Rust code → cargo run → stdout B
assert!(A == B)
```

**適用戦略（コンパイラ/トランスパイラ特有）**

```
E2E テスト対象の優先順位:
  1. サイレントな意味変更（実行結果が変わるが型エラーにならない）
     例: switch文のcase変数束縛の誤変換
  2. 副作用のある変換（console.log順序、例外発生タイミング）
  3. 数値計算の精度（i32 vs f64の境界）
  4. 文字列処理の等価性（Rust &str vs TS string の微妙な差）
```

---

### 4.4 パターンマッチの網羅性テスト

**定義と目的**

コンパイラ/トランスパイラは入力ASTのパターンマッチに強く依存する。Rustのパターンマッチは型システムによって網羅性（exhaustiveness）が保証されるが、「変換すべきASTノード種別をすべてハンドリングしているか」は静的には保証されない。

**Rust の exhaustiveness check の活用**

```rust
// Rust コンパイラが網羅性を強制する
match expr {
    Expr::Lit(_) => ...,
    Expr::Ident(_) => ...,
    // 新しいバリアントを追加すると、ここでコンパイルエラー → 対応強制
}

// `_` を使うと網羅性チェックをバイパスしてしまう（注意）
match expr {
    Expr::Lit(_) => ...,
    _ => emit_unsupported(), // ← 新しいバリアントが気づかずスキップ
}
```

**テスト戦略**

- SWC ASTの全バリアントリストを定期的に確認（SWC バージョンアップで増加する可能性）
- `_` ワイルドカードが使われている箇所に対し、「通過するだけのテスト」を作成
- 各ASTノード種別に対応するフィクスチャが存在するかのチェックリスト管理

---

### 4.5 エッジケースの体系的列挙

**定義と目的**

同値分割とエラー推測を組み合わせ、コンパイラ固有のエッジケースを体系的に列挙する。

**エッジケース分類**

```
カテゴリ1: 空・最小
  - 空ファイル
  - 型注釈なし（型推論のみ）
  - 空の関数ボディ
  - 空の interface / class
  - 空の enum

カテゴリ2: 最大・複雑
  - 深くネストされた型（Option<Vec<HashMap<String, T>>>）
  - 長い関数（100+引数ではないが、大きなスイッチ文）
  - 多重継承相当（Mixin パターン）
  - 大規模なユニオン型（20+バリアント）

カテゴリ3: 境界・特殊値
  - 数値: 0, -0, NaN, Infinity, -Infinity, MAX_SAFE_INTEGER
  - 文字列: "", " ", "\n", "null", "undefined", 絵文字
  - boolean: true, false の両方

カテゴリ4: 相互作用
  - ジェネリックとオプショナルの組み合わせ
  - クロージャとミュータブル変数のキャプチャ
  - async/await とエラーハンドリングの組み合わせ
  - デコレータとクラス継承の組み合わせ

カテゴリ5: 構文の境界
  - セミコロンあり vs なし
  - trailing comma あり vs なし
  - アロー関数 vs function宣言（意味的等価性）
  - var vs let vs const（スコープの差異）
```

## 5. テストケース設計の品質基準

---

### 5.1 ISTQB 標準テスト設計プロセス

ISTQB CTFL v4.0 が定めるテスト設計の流れ:

```
1. テスト条件の特定
   ↓ (テストベースの分析)
2. テストケースの設計
   ↓ (技法の適用)
3. テストデータの特定
   ↓ (入力値・期待結果の具体化)
4. テスト環境の特定
   ↓ (実行条件の明確化)
5. テスト手順の定義
```

**テスト条件（Test Condition）の定義**

テストすべき事項（アスペクト、特性、状態遷移等）。フィクスチャのファイル名がテスト条件に対応（例: `type-narrowing.input.ts` → 「型narrowingが正しく変換されること」）。

---

### 5.2 独立性（Test Independence）

**定義**

各テストケースは他のテストケースに依存せず、単独で実行・失敗・成功できなければならない。

**適用ルール**

- テスト間で状態を共有しない（グローバルな型レジストリ等の副作用に注意）
- 実行順序が変わっても結果が変わらない
- 1テストケースの失敗が他のテストに影響しない

**本プロジェクトでの実装**

- `COMPILE_LOCK: Mutex<()>`: コンパイルテストを直列化（共有ファイルへの競合書き込み防止）
- `E2E_LOCK: Mutex<()>`: E2Eテストを直列化
- `snapshot_test!` マクロ: 各テストは独立したフィクスチャファイルを持つ

**注意点**

Mutex による直列化はテスト間の干渉を防ぐが、**テストの独立性が損なわれているサイン**でもある。コンパイルチェックが共有ディレクトリを必要とする現状は技術的制約として受け入れられているが、将来的には各テストが専用の一時ディレクトリを使う改善が望ましい。

---

### 5.3 最小性（Test Minimality / 1テスト1観点）

**定義**

各テストケースは1つの観点（テスト条件）のみを検証する。複数の観点を1つのテストに詰め込まない。

**メリット**

- 失敗時の原因特定が容易
- メンテナンスコストの低減
- テストドキュメントとしての読みやすさ

**適用指針（本プロジェクト）**

```
良い例: 1フィクスチャ = 1概念
  type-narrowing.input.ts → 型narrowingのみをテスト
  optional-chaining.input.ts → optional chainingのみをテスト

悪い例: 1フィクスチャに複数概念を詰め込む
  everything.input.ts → 全機能を1ファイルでテスト
  → 失敗時に「どの機能のバグか」が不明
```

**ただし例外あり**

統合的なフィクスチャ（`mixed.input.ts`等）は、機能間の相互作用をテストする目的で設計されており、「1テスト1観点」の原則外で意図的に複数概念を含む。

---

### 5.4 再現性（Reproducibility）

**定義**

同一の入力・環境で常に同一の結果を出すこと。テストは決定論的でなければならない。

**非決定性の原因と対策**

| 原因 | 影響 | 対策 |
|------|------|------|
| ファイルシステムのmtime競合 | カーゴのフィンガープリント誤判定 | `write_with_advancing_mtime()` でmtimeを単調増加 |
| 並行テストの競合 | コンパイルチェックの誤結果 | `COMPILE_LOCK`, `E2E_LOCK` で直列化 |
| HashMap の非決定的順序 | 生成コードの出力順序が不安定 | BTreeMap 使用 or ソート後に出力 |
| 外部サービスへの依存 | ネットワーク状況で変化 | モック化 or テストでの外部依存排除 |

---

### 5.5 トレーサビリティ（Traceability）

**定義**

テストケースとテスト対象（仕様、要件、バグ票）の対応関係が追跡可能であること。

**本プロジェクトでの実装**

- TODO・backlog の「I-XXX: 説明」番号 → 対応フィクスチャ名の命名規則
- フィクスチャ名が変換機能の「仕様書」として機能（`type-narrowing.input.ts` 等）
- `tests/snapshots/` のスナップショットファイルが「期待動作の記録」として機能

---

## 6. 技法間の補完関係マトリクス

| 技法 | 強み | 弱み | 最適補完技法 |
|------|------|------|------------|
| 同値分割 | 効率的な代表値選択 | 境界バグを見逃す | 境界値分析 |
| 境界値分析 | 境界バグを重点テスト | 組み合わせを見ない | デシジョンテーブル |
| デシジョンテーブル | 複合条件の網羅 | 組み合わせ爆発 | ペアワイズ |
| 状態遷移 | 動的挙動の網羅 | 大規模FSMに不向き | 同値分割（状態ごと入力） |
| ペアワイズ | 組み合わせ爆発の緩和 | 3-way相互作用見逃し | エラー推測（高リスク組み合わせ） |
| エラー推測 | 経験的な欠陥発見 | 体系性なし | チェックリストベース |
| C0（命令網羅） | 基本的な網羅確認 | 最弱 | C1（分岐） |
| C1（分岐） | 分岐の網羅 | 複合条件の深さ不足 | MC/DC |
| MC/DC | 条件の独立性証明 | 設計コスト高 | スナップショット（全体確認） |
| パス網羅 | 最強のホワイトボックス | 実用不可能な場合多 | ベーシスパステスト |
| データフロー | 変数ライフサイクル | ツール支援が必要 | 静的解析（clippy） |
| スナップショット | 出力の完全比較 | 「正しさ」は手動確認 | コンパイルテスト + E2E |
| 探索的テスト | 未知バグの発見 | 再現性・体系性なし | スクリプテッドテスト（回帰化） |

## 7. 本プロジェクト（ts_to_rs）への適用指針

### 現状のテスト層と適用技法の対応

```
Layer 1: integration_test.rs (スナップショット)
  └─ 技法: スナップショット/Golden Master
  └─ カバレッジ: AST変換パターンの出力形状
  └─ 欠陥検出: 変換ロジックの回帰

Layer 2: compile_test.rs (コンパイル確認)
  └─ 技法: 実装フィードバックループ
  └─ カバレッジ: 生成Rustの型安全性
  └─ 欠陥検出: 型エラー、構文エラー

Layer 3: e2e_test.rs (実行等価性)
  └─ 技法: 差分テスト（Differential Testing）
  └─ カバレッジ: 意味的等価性
  └─ 欠陥検出: サイレントな意味変更（最重要）

Layer 4: cli_test.rs (CLIインターフェース)
  └─ 技法: ユースケーステスト
  └─ カバレッジ: CLIのコマンド引数・出力形式
```

### 推奨される追加テスト戦略

**優先度 高（即時適用推奨）**

1. **チェックリストベース: ASTノード種別網羅確認**
   - SWC ASTの全Expr/Stmt/Declバリアントに対応するフィクスチャが存在するか確認
   - 欠落しているノード種別 → 新規フィクスチャ追加

2. **デシジョンテーブル: 型変換ルールの組み合わせ**
   - オプショナリティ × 型種別 × 可変性の組み合わせ表を作成
   - 未テストの組み合わせセルを発見 → フィクスチャ追加

3. **エラー推測: TypeScript固有の罠に対するE2Eテスト**
   - NaN, undefined, null, type coercion 等のE2Eフィクスチャ追加

**優先度 中（機能追加時に適用）**

4. **境界値分析: 数値型変換の境界**
   - i32::MAX/MIN, f64::NAN, f64::INFINITY 等の境界値フィクスチャ

5. **状態遷移テスト: TypeCollectorのフェーズ遷移**
   - 前方参照、循環参照のE2Eテスト強化

**優先度 低（成熟段階で適用）**

6. **ペアワイズ: 型特性の組み合わせ**
   - PICTツールで組み合わせテストケースを自動生成

7. **MC/DC: 型解決の複雑条件**
   - TypeResolver の複合条件ロジックに対するユニットテスト

### テストケース設計の実践指針（本プロジェクト固有）

1. **フィクスチャ命名規則**: `{技法-対象}-{エッジケース}.input.ts`
   例: `union-type-null-boundary.input.ts`, `generic-nested-optional.input.ts`

2. **E2Eテストの優先**: サイレントな意味変更（conversion-correctness-priority.mdのTier 1）を検出できるのはE2Eのみ。新規変換機能には必ずE2Eスクリプトを追加。

3. **スナップショットの更新フロー**: `cargo insta review` で変更を必ず目視確認。機械的に全承認しない。

4. **Honoベンチマークの活用**: `./scripts/hono-bench.sh` + `scripts/inspect-errors.py` は実質的に大規模探索的テスト。定期実行とカテゴリ分析がチェックリスト更新の入力になる。
